      "text": "# Internal Game Engine Design\n\nThe pandya.ai platform features a custom-built, turn-based game engine in Go. While built natively for performance, the engine's core logic is highly extensible through a **Lua-based scripting environment**.\n\n## \ud83c\udfd7\ufe0f Core Architecture\n\nThe engine resides in `internal/engine/` and functions as a deterministic state machine. It handles network synchronization, player turns, and state persistence, while delegating specific game rules to a **Lua VM**.\n\n### Key Components\n\n| Component | Description |\n| :--- | :--- |\n| **`GameDefinition`** | The complete bundle of a game, containing **Canvas JSON** (visuals) and **Lua Script** (logic). |\n| **`Ctx`** | The \"Context\" provided to Lua (current player, turn number, phase, RNG). |\n| **`State`** | The current snapshot of the game data (board, pieces, zones, variables). |\n| **`Engine`** | The Go runtime that manages the lifecycle of a session and executes Lua hooks. |\n\n## \ud83d\udd79\ufe0f Lifecycle of a Move\n\n```mermaid\nsequenceDiagram\n    autonumber\n    participant Go as Go GameManager / Service\n    participant LuaVM as Sandboxed Lua VM (gopher-lua)\n    participant HostAPI as Go Host API bindings\n    participant State as In-Memory State\n\n    Go->>LuaVM: Load game state variables & piece lists\n    Go->>LuaVM: Invoke on_move(ctx, move)\n    Note over LuaVM: Validate move conditions (e.g., cell distance, owner)\n    \n    rect rgb(30, 40, 60)\n        Note over LuaVM, HostAPI: Query or mutate board state via Go callbacks\n        LuaVM->>HostAPI: get_piece(piece_id)\n        HostAPI-->>LuaVM: piece properties (position, owner)\n        LuaVM->>HostAPI: move_piece(piece_id, target_cell)\n        HostAPI->>State: Update target cell state\n        HostAPI-->>LuaVM: success\n    end\n\n    LuaVM->>HostAPI: end_turn()\n    HostAPI->>State: Increment turn, swap active player index\n    \n    LuaVM-->>Go: return result (status code, error details)\n    Go->>Go: Run check_win() to update game status (WIN/DRAW/IN_PROGRESS)\n```\n\n1.  **Request**: A player sends a `MakeMove` RPC with an `actionId` and arguments (e.g., `from`, `to`, `pieceId`).\n2.  **Lua Execution**: The engine calls the `on_move(ctx, move)` function within the game's Lua script.\n3.  **Validation**: Inside `on_move`, the script validates the move against game rules (e.g., \"Is the path clear?\").\n4.  **State Mutation**: If valid, the Lua script mutates the game state directly.\n5.  **Hooks**: The engine automatically triggers Lua hooks like `on_turn_end`, `on_phase_change`, or `check_win_condition`.\n6.  **Broadcast**: The updated state is serialized and streamed to all connected clients via ConnectRPC.\n\n## \ud83d\udcdc Scripting with Lua\n\nDefining a game no longer requires recompiling the Go backend. Instead, the engine provides a rich API to Lua scripts:\n\n- **`ctx.state`**: Access and modify any part of the game state (pieces, zones, variables).\n- **`ctx.random`**: Use a seeded, deterministic PRNG for die rolls or shuffling.\n- **`ctx.current_player`**: Identify whose turn it is.\n- **`ctx.log`**: Structured logging for debugging game logic.\n\n### Advanced Mechanics API\n- **Programmed Movement**: Use `game.queue_action(player_index, move)` and `game.execute_queue()` to support deferred or simultaneous execution.\n- **Turn Mutations**: Explicitly alter the standard round-robin rotation via `game.set_current_player(player_index)`.\n\n### Example Lua Hook (Move)\n```lua\nfunction on_move(ctx, move)\n    if move.action_id == \"move_piece\" then\n        local piece = ctx.state.get_piece(move.piece_id)\n        -- Validation logic...\n        piece.position = move.to\n        ctx.end_turn()\n    end\nend\n```\n\n## \ud83d\udd12 State Visibility & Redaction\n\nTo support \"fog of war\" or hidden information (like cards in Poker, or hidden unit movement), the engine uses a `PlayerView` mechanism. \n\n- **Property-based Redaction**: Simply set a piece's property via `piece.set_attr(piece_id, \"hidden\", true)`. The Engine will automatically redact the `position` of this piece for all opponents, providing native Hidden Movement support.\n- **Scripted Filtering**: Before broadcasting state, it can run a Lua script to redact sensitive fields based on the recipient's `player_id`.\n\n## \u26a1 Interrupts & Reactions\nThe engine natively supports out-of-turn play via the `Interrupt` flag on action definitions. By setting `interrupt: true`, the standard \"not your turn\" engine check is bypassed, enabling Magic: The Gathering style reaction instances or counter-spells.\n\n## \ud83e\udde9 Custom Games (Game Authoring)\n\nThe **Game Author Service** (`internal/gameauthor`) bridges the gap between the visual canvas and the logic engine:\n1.  **Canvas JSON**: Defines zones, layouts (Grid, Hex, Deck), and piece templates.\n2.  **Logic Generation**: AI agents translate natural language rules into the imperative Lua script described above.\n3.  **Dynamic Loading**: When a game starts, the engine loads the Canvas and Script dynamically, allowing for infinite game variety without server restarts.\n\n### Paths API\nThe `paths` API is injected by the Host into the Lua VM to interact with predefined path configurations from the GDL canvas.\n- `paths.advance(path_id, current_cell, steps)`: Advances `steps` nodes along a given `path_id` starting from `current_cell`. If `current_cell` is not on the path, it returns `current_cell`. If it reaches the end of the path, it returns the final node.\n- `paths.get_nodes(path_id)`: Returns an array of cell coordinates representing the path.\n- `paths.get_node(path_id, index)`: Returns the cell coordinate string (e.g. `\"x,y\"`) at the given 1-based `index` in the path. Returns `nil` if the index is out of bounds or the path does not exist. This is useful for fetching starting or physical locations without hardcoding arrays in Lua.\n- `paths.get_index(path_id, cell)`: Returns the 1-based index of the `cell` string in the path. Returns `0` if the cell is not found on the path.\n"
